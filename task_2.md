# Задание № 2. Инструменты криптографии

## 1. Разница между кодированием, хешированием и шифрованием

*   **Кодирование** — это процесс преобразования данных из одного формата в другой по заранее известному, публичному алгоритму. **Цель** — обеспечить корректную передачу или хранение данных, а не их скрытие. Данные легко обратимы при знании алгоритма.
    *   **Пример в вебе:** Кодирование данных в формат Base64 для передачи бинарных файлов (например, картинок) в теле JSON-запроса или для встраивания небольших изображений прямо в HTML/CSS (data-URI).

*   **Хеширование** — это необратимое (одностороннее) криптографическое преобразование входных данных произвольной длины в строку фиксированной длины (хеш). **Цель** — проверить целостность данных или безопасно хранить пароли. Идеальный хеш невозможно обратить для получения исходных данных.
    *   **Пример в вебе:** Хранение паролей пользователей в базе данных. При регистрации от пароля вычисляется хеш (например, с помощью bcrypt), который и сохраняется. При следующем входе хеш от введенного пароля сравнивается с хранимым. Сам пароль никогда не хранится.

*   **Шифрование** — это обратимое криптографическое преобразование данных с использованием секретного ключа (или пары ключей). **Цель** — обеспечить конфиденциальность. Зашифрованные данные могут быть расшифрованы обратно в исходные только при наличии правильного ключа.
    *   **Пример в вебе:** Защита соединения между браузером пользователя и сервером с помощью протокола HTTPS (TLS), который использует шифрование для защиты всего передаваемого трафика от перехвата.

## 2. Симметричное vs. Асимметричное шифрование для менеджера

**Краткая сводка:**

| Аспект                | Симметричное шифрование                                        | Асимметричное шифрование                                       |
| :-------------------- | :------------------------------------------------------------- | :------------------------------------------------------------- |
| **Определение**       | Использует **один и тот же ключ** для шифрования и расшифровки. | Использует **пару ключей**: открытый (public) и закрытый (private). Что зашифровано одним, может быть расшифровано только другим. |
| **Аналогия**         | Замок с одним ключом. Один ключ и запирает, и отпирает дверь. Тот, у кого есть ключ, может и отправить сообщение в помещение, и прочитать его. | Почтовый ящик с щелью (публичный ключ) и замком с ключом (приватный ключ). Любой может опустить письмо через щель, но достать его может только владелец, у которого есть ключ от ящика. |
| **Плюсы**            | **Очень быстрое**, требует меньше вычислительных ресурсов. Идеально для шифрования больших объемов данных (например, всего сеанса связи). | Решает проблему **безопасной передачи ключа**. Открытый ключ можно свободно раздавать. Обеспечивает цифровые подписи. |
| **Минусы**           | **Проблема безопасной передачи ключа** тому, с кем нужно обмениваться данными. Если ключ перехватят, безопасность скомпрометирована. | **Медленное**, требует значительных вычислительных ресурсов. Неэффективно для шифрования больших данных. |

** Аналогия с замком/ключом и почтовым ящиком выбрана потому, что она интуитивно понятна и наглядно демонстрирует фундаментальное различие: в симметричном шифровании один секретный инструмент для всех операций, в асимметричном — публичная часть для одностороннего действия (отправка/шифрование) и приватная для доступа (получение/расшифровка). Это легко представить даже без технического бэкграунда.

## 3. "Черный" и "белый" списки алгоритмов

**"Черный" список (слабые, нерекомендуемые алгоритмы):**

*   **Шифрование:**
    1.  **DES (Data Encryption Standard)** — длина ключа (56 бит) мала для современных вычислительных мощностей. Уязвим к атаке **полного перебора (brute-force)**.
    2.  **RC4 (Rivest Cipher 4)** — имеет значительные статистические смещения в выходном ключевом потоке, что делает его уязвимым к **атакам на потоковое шифрование**. Быстро взламывается.
    3.  **Шифры в режиме ECB (Electronic Codebook)** (например, AES-ECB) — не используют вектор инициализации (IV), что приводит к тому, что одинаковые блоки открытого текста шифруются в одинаковые блоки шифротекста. Это позволяет проводить **анализ шаблонов** данных.
*   **Хеширование:**
    1.  **MD5** — крайне уязвим к **коллизиям** (возможность создать два разных файла с одинаковым хешем). Также уязвим к атакам **предвычислений (rainbow tables)**. Непригоден для защиты.
    2.  **SHA-1** — также считается криптографически нестойким из-за возможности генерации **коллизий** с приемлемыми для злоумышленника вычислительными затратами.
    3.  **Неиспользование "соли" (salt) при хешировании паролей** (даже с SHA-256) — делает пароли уязвимыми к атакам с помощью **радужных таблиц (rainbow tables)**.

**"Белый" список (рекомендуемые, стойкие алгоритмы):**

*   **Шифрование:**
    1.  **AES (Advanced Encryption Standard)** с длиной ключа **256 бит** в режимах **GCM** или **CBC** (при правильной реализации с уникальным IV). Признан стандартом во всем мире. Стойкость основана на сложности решения математической задачи, проверенной многолетним криптоанализом. Режим GCM также обеспечивает аутентификацию данных.
    2.  **ChaCha20-Poly1305** — современный потоковый шифр с аутентификацией. Эффективен на процессорах без аппаратного ускорения AES, часто используется в мобильных устройствах и в TLS. Доказанно стойкий и высокопроизводительный.
*   **Хеширование:**
    1.  **SHA-256 / SHA-384 / SHA-512** (из семейства SHA-2) — считаются криптографически стойкими, коллизии для них не найдены при существующих вычислительных мощностях. Широко применяются для проверки целостности и в блокчейне.
    2.  **Argon2, bcrypt, scrypt** — специальные **функции для хеширования паролей (KDF)**. Они не просто хешируют, а являются адаптивными, требуют значительных вычислительных ресурсов и памяти, что делает атаки перебором (brute-force) чрезвычайно медленными и дорогими. Argon2 — победитель конкурса PHC (Password Hashing Competition).

## 4. Тест-план для проверки базовой безопасности интернет-магазина электроники

### **Область 1: Вход в аккаунт (логин/пароль)**

**Что будем проверять и как:**

| **Что проверять (аспект безопасности)** | **Как проверять (конкретные шаги)** | **На что обращать внимание** |
|-------------------------|--------------------------------------|-----------------------------|
| **1. Защита от brute-force** | 1. Ввести неверный пароль 5 раз подряд для одного аккаунта.<br>2. Попробовать 6-ой раз. | • Появляется ли капча после 3-5 попыток?<br>• Блокируется ли аккаунт на время?<br>• Приходит ли уведомление пользователю о подозрительной активности? |
| **2. HTTPS передача** | 1. Открыть DevTools → вкладка Network.<br>2. Выполнить вход и найти POST-запрос с логином/паролем.<br>3. Проверить протокол и заголовки. | • URL начинается с `https://`<br>• В запросе нет передачи пароля в URL-параметрах<br>• В ответах нет заголовков, разрешающих HTTP (Strict-Transport-Security) |
| **3. Безопасность cookies** | 1. После входа открыть DevTools → Application → Cookies.<br>2. Найти сессионную куку. | • Флаги `HttpOnly` (защита от XSS)<br>• Флаг `Secure` (только по HTTPS)<br>• Флаг `SameSite` (Lax/Strict)<br>• Срок действия (не должен быть слишком долгим) |
| **4. Смена ID сессии** | 1. Запомнить значение сессионной куки до входа.<br>2. Выполнить вход и сравнить новое значение. | • После входа должна быть установлена **новая** сессионная кука<br>• Старая сессия должна стать недействительной |
| **5. Выход из системы** | 1. Войти в аккаунт.<br>2. Нажать "Выйти".<br>3. Попробовать перейти по прямой ссылке в ЛК. | • После выхода сессионная кука должна удаляться или становиться невалидной<br>• Сервер должен возвращать 401/403 на запросы к защищенным страницам |

**Инструменты:**
- **Burp Suite/OWASP ZAP**: Для перехвата и анализа запросов/ответов, проверки передачи данных.
- **Браузерные DevTools**: Для проверки cookies, анализа сетевых запросов.
- **Ручные тесты**: Создание тестовых аккаунтов с разными паролями.

**Что будем проверять и как:**

| **Что проверять (аспект безопасности)** | **Как проверять (конкретные шаги)** | **На что обращать внимание** |
|-------------------------|--------------------------------------|-----------------------------|
| **1. IDOR (доступ к чужим данным)** | 1. Войти под user1, получить order_id своего заказа (например, 100).<br>2. Под user2 получить order_id своего заказа (например, 101).<br>3. Под user1 попытаться получить доступ к заказу 101 через API или URL. | • Получаем ли мы данные чужого заказа?<br>• Какая ошибка возвращается? (Должна быть 403, а не 404 или 200)<br>• Проверить не только order_id, но и cart_id, user_id во всех запросах |
| **2. Целостность цен** | 1. Добавить товар в корзину.<br>2. Перехватить запрос и изменить цену на меньшую/отрицательную.<br>3. Отправить измененный запрос. | • Принимает ли сервер измененную цену?<br>• Проверяется ли цена на бэкенде при оформлении?<br>• Где хранится итоговая сумма — на клиенте или сервере? |
| **3. CSRF защита** | 1. Найти форму добавления в корзину или применения купона.<br>2. Проверить наличие скрытого поля с CSRF-токеном.<br>3. Попробовать отправить запрос без токена или с неверным. | • Есть ли CSRF-токен в форме?<br>• Проверяется ли токен на сервере?<br>• Истекает ли время жизни токена?<br>• Токен должен быть уникальным для сессии |
| **4. Логика бизнес-процессов** | 1. Добавить товар в корзину.<br>2. Изменить его количество на отрицательное или очень большое.<br>3. Попробовать применить один купон многократно. | • Обрабатывает ли система некорректные значения?<br>• Можно ли получить скидку больше 100%?<br>• Есть ли проверки на стороне сервера? |

**Инструменты:**
- **Burp Suite/OWASP ZAP**: Для перехвата и модификации запросов (изменение ID, цен, параметров).
- **Postman**: Для прямых запросов к API с измененными параметрами.
- **Собственные скрипты**: Для автоматизации проверки IDOR на разных ID.

### **Область 3: Оформление заказа**

**Что будем проверять и как:**

| **Что проверять (аспект безопасности)** | **Как проверять (конкретные шаги)** | **На что обращать внимание** |
|-------------------------|--------------------------------------|-----------------------------|
| **1. Безопасность платежей** | 1. Пройти до этапа оплаты.<br>2. Проверить, куда отправляются данные карты.<br>3. Проанализировать HTML-код платежной формы. | • Данные карты отправляются напрямую на сервер магазина или на шлюз?<br>• Есть ли iframe от платежной системы?<br>• Используется ли tokenization (замена данных на токен)?<br>• PCI DSS compliance (не хранятся ли данные карты) |
| **2. XSS в данных заказа** | 1. В поля "Имя", "Адрес", "Комментарий" ввести: `<script>alert(1)</script>`, `"><img src=x onerror=alert(1)>`<br>2. Оформить заказ и посмотреть, как данные отображаются: в ЛК, в письме, в админке. | • Выполняется ли скрипт?<br>• Экранируются ли спецсимволы?<br>• Проверять нужно везде, где данные отображаются |
| **3. SQL-инъекция в форме** | 1. В текстовые поля ввести: `' OR '1'='1`, `" OR ""="`, `1; DROP TABLE users`<br>2. Посмотреть на реакцию системы (ошибки, необычное поведение). | • Возникают ли SQL-ошибки в ответах?<br>• Меняется ли поведение приложения?<br>• Проверять нужно все поля ввода |
| **4. HTTPS на всех этапах** | 1. Пройти весь процесс оформления.<br>2. Проверить ВСЕ запросы в DevTools → Network. | • Все ли запросы идут по HTTPS?<br>• Нет ли mixed content (HTTP-ресурсы на HTTPS-странице)?<br>• Правильные ли заголовки безопасности? |
| **5. Валидация бизнес-логики** | 1. Попробовать оформить заказ на 0 рублей (через купоны, изменение цен).<br>2. Изменить итоговую сумму в запросе перед отправкой.<br>3. Попробовать выбрать несуществующий способ доставки. | • Проверяет ли сервер итоговую сумму?<br>• Можно ли обмануть систему скидок?<br>• Все ли решения принимаются на сервере? |

**Инструменты:**
- **Burp Suite Active Scan**: Для автоматического поиска уязвимостей в формах.
- **SQLMap**: Для автоматизированной проверки параметров на SQLi (с разрешения!).
- **XSS-чекеры**: Списки полезных нагрузок для XSS.
- **SSL-сканеры** (testssl.sh, SSL Labs): Для проверки конфигурации HTTPS.

---

### **Ключевые принципы при проверке:**
1. **Всегда проверяйте на стороне сервера** — любую логику, валидацию, проверку прав.
2. **Тестируйте с разными ролями** — пользователь, админ, неавторизованный гость.
3. **Проверяйте все каналы** — Web UI, мобильное приложение, API.
4. **Документируйте все шаги** — что делали, какой результат, уровень риска.
5. **Используйте легитимные тестовые данные** — никогда не тестируйте на production без явного разрешения.

## Инструменты для тестирования безопасности веб-приложений

**Для проведения проверок безопасности тестировщику помогут следующие категории веб-инструментов:

### 1. Инструменты для перехвата и анализа трафика (Прокси)
- **Burp Suite** (Community/Professional) — наиболее популярный комплексный инструмент. Позволяет перехватывать, анализировать и модифицировать HTTP/HTTPS трафик между браузером и сервером. Включает:
  - **Proxy** — для перехвата и изменения запросов/ответов
  - **Repeater** — для многократной отправки модифицированных запросов
  - **Intruder** — для автоматизации атак (перебор, фаззинг)
  - **Scanner** (в Pro версии) — для автоматического сканирования уязвимостей
- **OWASP ZAP (Zed Attack Proxy)** — бесплатный открытый аналог Burp Suite с похожим функционалом. Особенно хорош для начинающих.
- **Fiddler Classic** — бесплатный отладчик прокси для Windows, удобен для анализа трафика, но менее ориентирован на пентест.

### 2. Браузерные инструменты разработчика (DevTools)
Встроены во все современные браузеры (Chrome, Firefox, Edge):
- **Вкладка Network** — анализ всех сетевых запросов, проверка передачи данных по HTTPS, просмотр заголовков безопасности.
- **Вкладка Application/Storage** — просмотр и манипуляция с cookies, LocalStorage, SessionStorage (проверка флагов HttpOnly, Secure, SameSite).
- **Вкладка Console** — выполнение JavaScript, проверка клиентских уязвимостей.
- **Вкладка Security** (в Chrome) — быстрая проверка сертификатов и общих проблем безопасности.
- **Расширения браузера**:
  - **EditThisCookie** — удобное управление cookies.
  - **Wappalyzer** или **BuiltWith** — определение технологий сайта.
  - **Retire.js** — обнаружение устаревших JavaScript библиотек с уязвимостями.

### 3. Специализированные сканеры уязвимостей
- **Nikto** — сканер уязвимостей веб-серверов, проверяет опасные файлы, устаревшие версии ПО.
- **Nmap** с скриптами (`http-*`) — сканирование портов и обнаружение базовых уязвимостей веб-серверов.
- **SQLMap** — автоматизация тестирования на SQL-инъекции.
- **XSStrike** или **XSSer** — специализированные инструменты для поиска XSS.

### 4. Инструменты для тестирования API
- **Postman** или **Insomnia** — создание и отправка сложных запросов, тестирование авторизации, параметров.
- **SoapUI** — для тестирования SOAP и REST API с акцентом на безопасность.

### 5. Утилиты командной строки
- **cURL** — отправка кастомных HTTP-запросов с различными заголовками и параметрами.
- **OpenSSL** — проверка SSL/TLS конфигурации: `openssl s_client -connect example.com:443`
- **testssl.sh** — детальная проверка настроек SSL/TLS.
- **whatweb** — идентификация технологий на стороне сервера.

### 6. Инструменты для перебора и фаззинга
- **Dirb**, **Dirbuster**, **gobuster** — перебор директорий и файлов на сервере.
- **wfuzz** — фаззинг параметров веб-приложений.
- **Hydra** или **Medusa** — перебор паролей (только в легальных рамках тестирования!).

### 7. Платформы для практики и справочники
- **OWASP Web Security Testing Guide (WSTG)** — методическое руководство.
- **PortSwigger Web Security Academy** — бесплатные лаборатории для отработки навыков.
- **HackTheBox** или **TryHackMe** (веб-трэки) — практические задания в легальной среде.


**Проверка для ситуации: "Пользователь забыл разлогиниться на общественном компьютере"**

*   **Как это можно проверить?**
    1.  Залогиниться на приложении в браузере.
    2.  Закрыть вкладку браузера (но не сам браузер полностью), имитируя уход пользователя.
    3.  Открыть новую вкладку и напрямую перейти по URL личного кабинета (например, `/my/orders`). Также попробовать открыть сайт магазина с главной страницы.
    4.  Проверить, остается ли пользователь залогиненным и имеет ли доступ к конфиденциальным данным без повторного ввода пароля.

*   **Что должно происходить для защиты?**
    1.  **Истечение срока действия сессии на сервере** после короткого периода бездействия (15-30 минут).
    2.  **Использование короткоживущих access-токенов** (например, JWT с expiration time) вместо бессрочных сессий.
    3.  **Наличие функции "Выйти со всех устройств"** в настройках безопасности аккаунта.
    4.  **Вывод информационного сообщения** при входе в аккаунт: "Вы зашли с нового устройства/браузера [дата, время]. Рекомендуем выходить из аккаунта на общественных компьютерах".
    *   **Идеальная защита:** Критические действия (просмотр персональных данных, изменение пароля, подтверждение заказа) должны требовать **повторной аутентификации** (re-authentication), даже если сессия активна.
